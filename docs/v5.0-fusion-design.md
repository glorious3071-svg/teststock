## 一、为什么要融合

20 年的纸上推演沉淀了**方法论灵魂**，但缺乏**工程基础设施**。
teststock 项目恰好提供了**工程基础设施**，但缺乏**方法论灵魂**。

| 维度 | 20 年探索（灵魂） | teststock（骨架） | 融合后 v5.0 |
|---|---|---|---|
| 数据库 | SQLite 单表 + tushare 客户端 | **MySQL 12 张宏观表 + Podman 部署** | **采用 teststock 的 MySQL，扩展信号表** |
| 战略层 | 年度战略书（手写 Markdown） | **LLM Agent 多轮对话生成** | **LLM Agent 生成 + 评分卡量化校验** |
| 战术层 | **v4.0 v3 反应式纪律触发器** | 无 | **以代码实现触发器引擎** |
| 评分机制 | **v3.4 双向评分卡 / v3.6 动能转换** | v3.6 仅有文档无代码 | **完整实现双层评分** |
| 回测引擎 | 手工脚本逐年回测 | **buy-and-hold / equal-weight / strategic** | **扩展支持触发器逐月回测** |
| 多年回测 | 8-20 年穿越 | **单年回测** | **多年穿越 + 跨周期统计** |
| 防上帝视角 | 凭良心约束 | **as_of_date 强制截断** | **沿用 teststock 的强约束** |
| 决策记录 | Markdown 文档 | **JSON Session 文件** | **JSON + 触发器历史表** |

**核心判断**: teststock 是**优秀的工程底座**，不要重写。我们要做的是**在其上加层**。

## 三、v5.0 三层决策架构

```
┌───────────────────────────────────────────────────────────────┐
│  Layer 1: 战略层 (Annual Direction)                            │
│  ─ 频率: 每年初一次                                            │
│  ─ 输入: 上一年 12-31 截止的宏观数据 + ETF 池                  │
│  ─ 决策者: LLM Agent + 评分卡量化校验                          │
│  ─ 输出: 年度配置 JSON (权益仓位 + ≤5 只 ETF 主题)              │
│  ─ 沿用: teststock/agents/annual_direction (LLM 部分)         │
│  ─ 扩展: 增加评分卡一致性检查                                  │

                              ↓

│  Layer 2: 战术层 (Tactical Response) — 新增                    │
│  ─ 频率: 每月末一次                                            │
│  ─ 输入: 当月已知的 PE/价格/政策事件                            │
│  ─ 决策者: 触发器引擎 (确定性规则，不调用 LLM)                  │
│  ─ 输出: A 再平衡 / B 政策加仓 / C 减仓 / D 加仓 / 无动作       │
```

### 三层职责清晰边界

| 决策类型 | 责任层 | 频率 | 决策方式 |

| 估值过热减仓 | 战术层 | 月度 | C 触发器 |
| 估值过低加仓 | 战术层 | 月度 | D 触发器 |
| 政策事件加仓 | 战术层 | 月度 | B 触发器 |
| 偏离再平衡 | 战术层 | 月度 | A 触发器 |

## 四、触发器引擎 (核心 — 新增)

### 4.1 触发器规则定义（沿用 v4.0 v3）

```python
@dataclass
class TriggerRule:
    rule_id: str         # 'A' / 'B' / 'C' / 'D'
    name: str
    condition: Callable  # signals: dict → bool
    action: Callable     # portfolio, signals → action_spec
    lock_months: int = 3
```

### 4.2 v4.0 v3 四大触发器（反应式纪律）

#### A. 再平衡触发器
- **条件**: 任一 ETF 实际权重偏离目标 ±5pp
- **动作**: 卖超买不足，恢复目标权重
- **锁定**: 无锁定期，可月月触发

#### B. 政策触发器
- **条件**: PE_TTM < 12 且当月有央行降准/降息事件
- **动作**: 现金的 30% 加仓至当时低估行业 ETF
- **锁定**: 触发后 3 个月对称锁

#### C. 减仓触发器
- **条件**: PE_TTM > 16 且 (单月涨幅 > 15% 或近 3 月累计 > 25%)
- **动作**: 减仓 10pp 至现金
- **锁定**: 3 个月对称锁

#### D. 加仓触发器
- **条件**: PE_TTM < 11 且 (单月跌幅 > 10% 或近 3 月累计 < -20%)
- **动作**: 现金的 50% 加仓
- **锁定**: 3 个月对称锁

### 4.3 锁定期与对称性
- 任一触发器被触发后，**同向**触发器在 3 个月内不能再被触发
- 反向触发器可立即触发（即 C 触发后，D 可立即触发，反之亦然）

### 4.4 数据库表 (`sql/trigger_schema.sql`)

```sql
-- 触发器规则
CREATE TABLE trigger_rule (
    rule_id        VARCHAR(8) PRIMARY KEY,
    name           VARCHAR(64),
    condition_expr TEXT,    -- 可读规则描述
    action_expr    TEXT,
    lock_months    INT DEFAULT 3,
    is_active      TINYINT(1) DEFAULT 1,
    version        VARCHAR(16) DEFAULT 'v4.0_v3'
);

-- 触发器执行历史
CREATE TABLE trigger_history (
    history_id     BIGINT AUTO_INCREMENT PRIMARY KEY,
    trade_date     DATE,
    year_month     CHAR(6),
    rule_id        VARCHAR(8),
    triggered      TINYINT(1),
    signal_snapshot JSON,
    action_taken   JSON,
    portfolio_before JSON,
    portfolio_after JSON,
    is_backtest    TINYINT(1)
);

-- 触发器锁定期
CREATE TABLE trigger_lock (
    rule_id         VARCHAR(8),
    direction       VARCHAR(8),  -- 'reduce' / 'add'
    locked_until_ym CHAR(6),
    PRIMARY KEY (rule_id, direction)
);
```

## 五、评分卡引擎 (v3.4 双向化 — 新增)

### 5.1 评分卡设计原则
**双向化**: 每个维度同时给出"看多分"和"看空分"，避免单向偏见。

### 5.2 五大维度评分（每维度 -5 ~ +5）

| 维度 | 看多信号 | 看空信号 |
|---|---|---|
| 估值 | PE < 11, PB < 1.3 | PE > 16, PB > 2.0 |
| 流动性 | 央行降准/降息，M1 上升 | 央行加息，M1 下降 |
| 基本面 | PMI > 51 持续上升，工业增加值 > 6% | PMI < 49 连续 3 月，企业利润下滑 |
| 情绪 | 两融余额下降，新基发行冷淡 | 两融激增，新基爆款连发 |
| 政策 | 中央定调"稳增长""活跃资本市场" | 印花税上调，监管收紧 |

### 5.3 综合评分映射至战略

| 综合分 | 权益仓位 | 现金 |
|---|---:|---:|
| ≥ +15 | 80% | 20% |
| +5 ~ +15 | 70% | 30% |
| -5 ~ +5 | 60% | 40% |
| -15 ~ -5 | 50% | 50% |
| ≤ -15 | 40% | 60% |

### 5.4 战略层 vs 评分卡的关系
- **LLM 给出年度配置建议** → **评分卡作为一致性校验**
- LLM 建议 80% 仓位但评分卡显示 50%，触发**质疑信号**，要求 LLM 解释或调整

## 六、动能转换器 (v3.6 — 新增实现)

### 6.1 核心命题
**外部信号需要被市场内部状态"转换"后才能决定行情**。

同一个"PE 过热"信号:
- 在牛市主升浪初期 → 被吸收，市场继续涨
- 在主升浪末期 → 被放大，触发顶部

### 6.2 四个转换维度

```python
@dataclass
class MomentumTransformer:
    # A: 政策预期 price-in 程度 (0.3 - 2.0)
    policy_priced_in: float
    
    # B: 资金性质粘性 (0.7 - 2.0)
    capital_stickiness: float
    
    # C: 牛市动能阶段 (0.5 - 1.5)
    bull_phase: float
    
    # D: 估值-盈利赛跑节奏 (0.5 - 2.0)
    valuation_earnings_race: float
    
    def transform(self, raw_signal_score: float) -> float:
        # 综合转换率
        rate = (self.policy_priced_in
                * self.capital_stickiness
                * self.bull_phase
                * self.valuation_earnings_race) ** 0.25
        return raw_signal_score * rate
```

### 6.3 用法
- 评分卡先算原始分（看多 - 看空）
- 动能转换器对原始分做加权调整
- 用调整后的分驱动战略仓位决策

## 七、多年穿越回测引擎 (新增)

### 7.1 回测循环
```python
def run_multi_year_backtest(start_year=2006, end_year=2025, initial_cash=1_000_000):
    portfolio = Portfolio(cash=initial_cash)
    history = []
    
    for year in range(start_year, end_year + 1):
        # 1. 年初: 战略层决策
        allocation = annual_direction_agent.decide(year, mode='backtest')
        scorecard_check = scorecard.evaluate(year)
        if scorecard_check.disagrees_with(allocation):
            allocation = scorecard.adjust(allocation)
        portfolio.set_target_allocation(allocation)
        
        # 2. 月度循环: 战术层反应式纪律
        for month in range(1, 13):
            signals = collect_monthly_signals(year, month)
            for rule in [A_rule, B_rule, C_rule, D_rule]:
                if rule.condition(signals, portfolio):
                    action = rule.action(portfolio, signals)
                    portfolio.execute(action)
                    trigger_history.log(rule, signals, action)
            portfolio.mark_to_market(end_of_month_prices)
            history.append(portfolio.snapshot())
    
    return BacktestReport(history)
```

### 7.2 输出报告
- 期末净值、CAGR、最大回撤
- 每年战绩 vs 沪深300 对比
- 触发器触发次数与有效性统计
- 跨极端市场表现：2008/2015/2018/2022 四次熊市

### 7.3 防上帝视角强约束
- 沿用 teststock 的 `as_of_date` 机制
- 战略层 LLM 仅看到上年 12-31 之前数据
- 战术层每月只看到当月末之前数据
- 触发器规则版本固化在 `trigger_rule` 表，回测时锁定 v4.0_v3 版本

## 九、设计原则的诚实

### 不能违背的红线（来自 20 年探索）
1. **不预言未来** — 触发器只能反应已发生事实，不能基于"预期央行会降息"
2. **不修改阈值救场** — 某年表现差不能调阈值（这是上帝视角）
3. **诚实接受牛市跑输** — 留 15-30% 现金的代价是普涨牛市跑输
4. **不预设核心资产** — ETF 池每年随评分卡而变，不抱团白酒/新能源

### 必须保留的工程纪律（来自 teststock）
1. **as_of_date 强约束** — 回测模式严禁未来信息
2. **数据 gaps 透明** — 缺失数据不掩饰
3. **决策可追溯** — 每次决策的输入数据 + 输出 + 时戳 全部入库
4. **LLM 输出结构化** — JSON 不能用自然语言糊弄
